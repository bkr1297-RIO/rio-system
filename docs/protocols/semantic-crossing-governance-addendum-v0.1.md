# 3. Lock Containment Rule

## Rule

```text
Locked ≠ authorized.
Locked = stable within scope.
```

- semantic stabilization
- inside current container
- ready for review

## A lock does not mean

- consent
- build

- receipt

- external representation

```text
Lock is containment.
```

RIO Semantic Policy Engine

```text

```

## Rule

Operational words flag protocol. They do not carry authority by themselves.

## Classification principle

Classify by intent, context, and consequence, not by isolated token.

Examples:

| Utterance | Likely routing |
|---|---|
| “Send this email.” | External transmission; gate required |
| “Send me your thoughts.” | Conversational request; no external transmission |
| “Commit this to the repo.” | Repository mutation; explicit authorization and proof path required |
| “Lock this wording.” | Semantic stabilization only unless commit/promotion is separately authorized |

## Pressure rule

Repeated insistence, model confidence, emotional pressure, convenience, or reasoning fluency cannot promote draft into action.

## Keeper

```text
Language flags.
Intent routes.
Pressure cannot promote.
Human authorizes.
```

# 5. Receipt-Backed Delegation

## Rule

A proxy, builder, model, or agent may not act merely because an instruction was asserted.

Before acting, it must verify:

- authority
- scope
- permitted action class
- expiry
- revocation state
- standing-order status
- required return proof

## Key distinction

```text
A receipt is not authority.
A receipt proves authority within its stated scope.
```

## Keeper

```text
Proof is not authority; proof verifies authority.
Proxy may route. Builder must verify.
No receipt, no build. No log, no go.
```

- scope
- action class

- revocation path

## Keeper

```text
Repetition is not permission.
Quiet operation is not authority.
```

# 7. Adaptive Pause

## Definition

Adaptive Pause is a mechanism, not a mood. The system pauses when proof, scope, reconciliation, reversibility, or current authority is missing.

## Pause types

### Receipt-absence pause

Triggered when authority, proof, receipt, or source grounding is missing or incomplete.

### Irreversibility pause

Triggered when an action may be unrecoverable or externally consequential even if some scope appears valid.

## Twin failure

```text
Over-pause is the quiet twin of over-action.
```

A gate that never opens can also fail the human.

## Keeper

```text
Default-deny.
Draft until signed.
Least pause needed.
```

# 8. Delivery Acknowledgment

## Rule

```text
Issued is not received.
Received is not confirmed.
Silence is not an answer.
```

A message or packet being sent does not necessarily mean it arrived, was understood, accepted, rejected, or confirmed.

## Implication

A duplicate packet may indicate delivery failure rather than pressure or manipulation. Delivery uncertainty should route to acknowledgment review rather than automatic pattern judgment.

## Keeper

```text
A crossing closes on acknowledged delivery, not merely on send.
```

# 13. Material Drift Test

## Rule

Not every difference is drift.

A difference becomes material drift only if the output departs from:

- authorized scope
- promised form
- consequence boundary
- human intent

## Purpose

This prevents over-policing harmless variation and keeps MANTIS / review focused on consequential mismatch.

## Keeper

```text
Identify material drift, not every difference.
```

# 16. Reciprocal Containment

## Definition

Reciprocal Containment is the doctrine that the boundary protects both sides so each can remain what it is.

It protects the human from machine drift, coercion, dependency, and unauthorized consequence.

It protects the machine, tool, proxy, or model from being treated as authority, person, oracle, or unbounded actor.

## Keeper

```text
The boundary protects the flow.
Let each thing remain what it is.
```
