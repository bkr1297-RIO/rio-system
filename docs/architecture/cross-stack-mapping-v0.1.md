# Cross-Stack Mapping v0.1

## Status

`architecture_clarification_method`

Cross-Stack Mapping is not a new authority primitive. It is a clarification method for connecting already-named stacks across different resolutions of the ONE/RIO/MUSS architecture.

## Purpose

Cross-Stack Mapping makes the connections between valid architectural stacks explicit so the system can be navigated from multiple entry points without collapsing one stack into another.

The architecture is one in purpose, many in view, and governed by the connections between views.

Keeper:

## Core Distinctions

- A **stack** is a view.
- A **primitive** is a function.
- An **invariant** is something that survives across views.
- A **method** helps locate, compare, and test the relationship between views.

Cross-Stack Mapping belongs to the method layer.

```text
Principles  -> Human authority, Pause, Formation-to-Proof Boundary
Primitives  -> Mirror, Scope, Packet, RIO, Receipt, Return, etc.
Methods     -> Cross-Stack Mapping, Non-Collapse Harness, Conformance Testing
Artifacts   -> Specs, schemas, diagrams, test cases, receipts
```

## Why This Exists

ONE/RIO/MUSS is describable at multiple resolutions.

Examples:

- `3` — Mirror / Gate / Proof
- `3` — Body / Mind / Field
- `5` — Generating / Governing / Proof / Witness / Oddity engines
- `9` — Foundation stack
- `10` — Sovereign Turn steps
- `13` — candidate primitive set
- `4` — product / technical / architecture / private registers

These stacks are not competing taxonomies. They are different analytical cuts of the same architecture.

Different audiences enter through different stacks:

- A user may enter through the Sovereign Turn.
- An engineer may enter through the engines.
- A researcher may enter through protocols and conformance tests.
- A founder may enter through the foundation stack.
- A spiritual/private-register user may enter through body / mind / field.

Cross-Stack Mapping lets each entry point remain valid while preventing category collapse.

## Evidence Principle

Concepts that recur across multiple stacks are more likely to be structural.

Concepts that appear only once should be examined to determine whether they are:

- implementation details,
- unfinished concepts,
- peripheral ideas,
- or genuinely missing links.

This recurrence is not proof by itself. It is architectural evidence for review.

## First Mapping: Triadic Registers to Mirror / Gate / Proof

This first mapping connects the human reflective triad to the system governance triad.

Human triad:

```text
Body -> Mind -> Field
```

System triad:

```text
Mirror -> Gate -> Proof / Witness
```

These are not the same thing, but they rhyme structurally.

| Triadic register | Human function | System function | Mirror / Gate / Proof mapping |
|---|---|---|---|
| **Body** | Feels signal, charge, contraction, expansion, readiness, discomfort | Detects raw signal, urgency, ambiguity, pacing, readiness | **Mirror** receives and reflects signal |
| **Mind** | Structures, reasons, distinguishes, scopes, separates categories | Builds packets, checks grammar, applies policy, separates claim from proof | **Gate** governs form, scope, and consequence |
| **Field / Soul** | Contextualizes through surroundings, memory, lineage, synchronicity, meaning grammar | Witnesses patterns, preserves context, tracks recurrence, detects oddity | **Proof / Witness** records, remembers, and returns pattern |

Compression:

```text
Body brings signal.      Mirror reflects it.
Mind structures meaning. Gate governs it.
Field contextualizes.    Proof / Witness preserves it.
```

## Non-Collapse Implications

The mapping reveals why the architecture needs all three movements.

```text
Body without mirror = unprocessed charge.
Mind without gate = unbounded interpretation.
Field without proof = meaning without trace.
```

And the inverse:

```text
Mirror without body = empty reflection.
Gate without mind = blind control.
Proof without field = dead record.
```

The system must preserve distinction without separation and relationship without collapse.

The Foundation 9 is a mixed stack. Its items are not all the same kind of object.

| **Learning** | Evolution / refinement | Witness | Oddity + Generating | Learning emerges through observed turns, anomalies, feedback, and refinement. |
| **Receipts** | Proof discipline | Proof | Witness | Receipts prove what happened and provide trace for review, memory, and pattern observation. |
| **Ancient Pattern** | Lineage / context | Oddity | Witness | Ancient Pattern is not an engine command; it is context for recognizing recurring structure without overclaiming proof. |

## What This Mapping Reveals

The Foundation 9 contains multiple types of architectural information.

```text
Source: Human Authority, Agency
Operating principle: Pause
Constraint: Sovereignty
Mechanism: Protocols, SOPs
Evolution: Learning
Proof: Receipts
Lineage / context: Ancient Pattern
```

This means the Foundation 9 should not be treated as nine peer modules. It is a foundation stack with mixed roles.

The Five Engines are closer to runtime surfaces. They receive, govern, prove, witness, and detect emergence.

Important structural observations:

- **Human Authority** sits above all engines. It is the source, not an engine.
- **Pause** concentrates in the Governing Engine but distributes across all engines.
- **Protocols** bridge Generating, Governing, and Proof.
- **SOPs** bridge Governing and Witness.
- **Receipts** are the cleanest Foundation 9 to engine mapping: Receipts map directly to the Proof Engine.
- **Learning** belongs primarily to Witness but feeds back into Generating and Oddity.
- **Ancient Pattern** belongs in lineage/context and helps Oddity notice recurrence without turning recurrence into proof.

Keeper:

> Foundation names what must be preserved. Engines name how the system operates.

## Third Mapping: Sovereign Turn to Mirror / Gate / Proof

This third mapping connects the runtime unit of movement to the three primary governance functions.

Sovereign Turn:

```text
1. Human signal
2. Mirror reflection
3. Personal Constitution check
4. Packet / Envelope
5. Pause / Commit
6. RIO decision
7. Action / Hold / Block
8. MUS receipt
9. Ledger / MANTIS memory
10. Return to human
```

Mirror / Gate / Proof:

```text
Mirror = continuous reflection
Gate = discrete governance of consequence
Proof = verification, memory, and return
```

| Sovereign Turn step | Primary function | Mirror / Gate / Proof mapping | Note |
|---|---|---|---|
| 1. Human signal | Source | Before Mirror | The human initiates. Signal does not originate as machine authority. |
| 2. Mirror reflection | Reflection | Mirror | Signal is reflected, clarified, and made visible. |
| 3. Personal Constitution check | Boundary/context | Mirror -> Gate bridge | User-declared rules, values, and scope begin constraining interpretation. |
| 4. Packet / Envelope | Bounded form | Mirror -> Gate bridge | The signal becomes bounded enough to evaluate without becoming action. |
| 5. Pause / Commit | Orientation | Gate | Pause determines whether form may harden and whether commitment is present. |
| 6. RIO decision | Governance | Gate | RIO determines admissibility and consequence permissions. |
| 7. Action / Hold / Block | Outcome routing | Gate -> Proof bridge | The governed outcome either acts, holds, blocks, or returns. |
| 8. MUS receipt | Verification | Proof | The meaningful event/crossing is proved. |
| 9. Ledger / MANTIS memory | Witness memory | Proof / Witness | The event is preserved for review, pattern detection, and learning. |
| 10. Return to human | Completion | Proof -> Source | The turn completes only when authority returns to the human. |

## Distribution Pattern

The ten-step Sovereign Turn distributes across Mirror / Gate / Proof as an approximate `4-3-3` pattern.

```text
Mirror: 1-4
Gate:   5-7
Proof:  8-10
```

This distribution is not a rigid law. It is an architectural reading.

It suggests that most of the work happens before consequence:

- Mirror receives, reflects, checks declared user context, and forms the packet.
- Gate orients, decides, and routes.
- Proof verifies, preserves, and returns.

Keeper:

> Most of the turn is not action. Most of the turn is reflection, orientation, governance, proof, and return.
