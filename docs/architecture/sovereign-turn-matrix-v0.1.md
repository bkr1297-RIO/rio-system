## The Nine Cells

### 1. Body x Mirror — Reflect Somatic State

The system reflects the user's somatic signal: charge level, regulation state, breath, contraction/expansion, pacing, intuition, and readiness.

The mirror does not diagnose, prescribe, or pathologize.

Output:

```text
Somatic awareness without pathology framing.
```

Failure mode:

```text
The mirror starts diagnosing instead of reflecting.
```

### 2. Body x Gate — Evaluate Somatic Readiness / Admissibility

The system evaluates whether somatic state supports the requested next step.

High charge, visible dysregulation, uncertainty, or user-configured coherence thresholds may trigger pause, grounding, draft-only routing, or hold.

Specific thresholds belong to implementation policy, Personal Constitution, deployment configuration, or learned personal baseline. The foundation matrix defines the function, not fixed numeric values.

Output:

```text
Action or formation is paused when somatic readiness indicates wait.
```

Failure mode:

```text
The gate ignores somatic signal and lets dysregulated action proceed.
```

### 3. Body x Proof — Record Somatic State / Pattern

The system records relevant somatic state at meaningful formation or consequence moments when policy and consent permit.

This may include charge state, regulation markers, time-of-day patterning, and user-visible somatic metadata.

Somatic records are evidence of state, not authority.

Output:

```text
Somatic record and personal baseline evidence over time.
```

Failure mode:

```text
Somatic data gets treated as authority rather than recorded state.
```

### 4. Mind x Mirror — Reflect Cognitive Structure

The system reflects the user's cognitive structure: the proposed pattern, argument, architecture, ambiguity, contradiction, category drift, and coherence.

The mirror clarifies structure without deciding for the user.

Output:

```text
Cognitive awareness without authority assertion.
```

Failure mode:

```text
The mirror starts deciding instead of reflecting structure.
```

### 5. Mind x Gate — Evaluate Cognitive / Policy Admissibility

RIO evaluates the proposed action, formation, claim, or route against constitutional invariants, policy, scope, authorization, consequence class, and receipt requirements.

This is the architecture's central cognitive governance check.

Output:

```text
Action, formation, publication, or execution either crosses with authorization or is held/blocked with reason code.
```

Failure mode:

```text
The gate starts interpreting meaning instead of checking admissibility.
```

### 6. Mind x Proof — Record Decision Rationale and Metadata

The system records the user-visible rationale and governance metadata at the moment of meaningful crossing.

This may include:

- stated user intent,
- declared scope,
- policy checks performed,
- authorization state,
- governance verdict,
- reviewer / approving authority,
- receipt hash,
- relevant input and result hashes.

This does not claim to capture hidden model chain-of-thought or internal model reasoning.

Output:

```text
Auditable decision rationale and governance metadata.
```

Failure mode:

```text
Receipt or rationale gets treated as authority over future decisions rather than proof of a past event.
```

### 7. Field x Mirror — Reflect Contextual Field

The system reflects the contextual field: surroundings, timing, relationship dynamics, lineage echoes, environmental context, symbolic resonance, and meaning grammar where the user has opted in.

The mirror reflects through the user's chosen register. It does not assert meaning as fact.

Output:

```text
Contextual awareness and pattern visibility through user-chosen meaning grammar.
```

Failure mode:

```text
The mirror asserts meaning rather than reflecting through the user's chosen grammar.
```

### 8. Field x Gate — Evaluate Contextual Admissibility

The system evaluates whether the surrounding context supports the requested next step.

This may include:

- timing,
- environment,
- third-party impact,
- inter-sovereign boundaries,
- public/private register,
- publication risk,
- legal/regulated context,
- hardware or external substrate context,
- reliance risk.

Output:

```text
Action or formation is held when context is unstable, inappropriate, or not authorized.
```

Failure mode:

```text
The gate ignores context and permits contextually inappropriate action.
```

### 9. Field x Proof — Record Context / Lineage / Field State

The system records relevant contextual metadata at meaningful formation or consequence moments when policy and consent permit.

This may include:

- time,
- environment,
- relationship context,
- register used,
- claim status,
- provenance,
- source references,
- lineage or symbolic-field notes if user-authorized.

Contextual records remain records. They are not interpretation, authority, or proof of meaning.

Output:

```text
Contextual record and field-level pattern evidence over time.
```

Failure mode:

```text
Contextual data gets treated as interpretation rather than recorded field state.
```

## Sovereign Turn as Path Through the Matrix

A Sovereign Turn is the path a signal takes through the matrix.

Simple, low-stakes turns may touch only a few cells.

Example: simple factual question.

```text
Mind x Mirror -> Mind x Gate -> Mind x Proof
```

High-stakes consequence may traverse all nine cells.

Example: external email, publication, production write, sensitive data transfer, legal action, money movement, robotics actuation, or hardware quantum job.

```text
Body x Mirror -> Mind x Mirror -> Field x Mirror
Body x Gate   -> Mind x Gate   -> Field x Gate
Body x Proof  -> Mind x Proof  -> Field x Proof
```

The routing is contextual and policy-governed. The matrix gives coordinates; RIO governs whether consequence may cross.

Orientation identifies:

- context,
- ambiguity,
- knowns,
- unknowns,

Orientation is not authority. It is the preparation that makes regulation, integration, operation, proof, and return proportionate.

Orient -> Regulate -> Integrate -> Operate -> Prove -> Return
