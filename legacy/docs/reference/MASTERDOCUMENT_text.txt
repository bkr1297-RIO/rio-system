SECTION 1: EXECUTIVE SUMMARY
WHAT WAS DISCOVERED
Security audit of production RIO gateway revealed THREE critical gaps and ONE usability issue:
1.

CRITICAL: Token Replay Possible
Execution tokens issued by /execute could be reused
Attacker could send same token multiple times
Impact: Action executed N times from single approval

2.

CRITICAL: Request Replay Possible
POST requests could be sent multiple times
No nonce or timestamp validation
Impact: Duplicate authorizations, duplicate executions

3.

CRITICAL: Ed25519 Signatures Optional
Authorization could be submitted without signature
Default mode was “optional”
Impact: Weak cryptographic enforcement

4.

MEDIUM: Intent Loss on Restart
In-memory intent storage (no persistence)
In-progress intents lost on gateway crash
Impact: Operational continuity issue (not security)

Fix #1: Token Burn + TTL File: gateway/security/token-manager.mjs Impact: Single-use tokens, 30-min TTL, prevents replay
Fix #2: Replay Prevention File: gateway/security/replay-prevention.mjs Impact: Nonce + timestamp validation, prevents duplicate requests
Fix #3: Ed25519 Required Files: gateway/security/oauth-hardened.mjs, authorize-hardened.mjs Impact: Cryptographic signatures mandatory on
all authorizations
Fix #4: Intent Persistence (TODO) Work: 1 hour, straightforward database integration

SECTION 2: ARCHITECTURE — THE FOUR FIXES
FIX #1: TOKEN BURN +
TTL (Single-Use Execution Tokens)

File: gateway/security/token-manager.mjs (140 lines)
PROBLEM

POST /execute issues an execution token to the agent. Agent executes action, calls POST /execute-confirm with token. BUT:

No validation that token is single-use.
Attack: Attacker obtains token, uses it 10 times → action executes 10x
SOLUTION
1. issueExecutionToken(intentId) → returns token + expiry
Token is random UUID
Stored with intent_id + issued_at + expires_at (30 min)
2. /execute-confirm calls validateAndBurnToken()
Checks token hash exists
Checks token is not burned
Checks token is not expired
If valid: MARK AS BURNED (can’t reuse)
If invalid: REJECT with FAIL CLOSED
3. Automatic cleanup

Expired tokens removed every 5 minutes
Burned tokens removed after retention period
SECURITY GUARANTEE

Each execution token can be used EXACTLY ONCE, for EXACTLY ONE INTENT. Attacker

cannot replay token. Attacker cannot use token for different intent.
PRODUCTION NOTE

Currently in-memory. For high-scale production, upgrade to Redis: $ npm install redis Then

update token-manager.mjs to use Redis (code ready, commented)
FIX #2: REPLAY
PREVENTION (Nonce + Timestamp Validation)

File: gateway/security/replay-prevention.mjs (170 lines)
PROBLEM

POST /intent, /govern, /authorize, /execute, /execute-confirm can be sent multiple times.

Attack #1: POST /authorize twice with same signature → Attacker approves same intent twice → Second approval might be after first authorization
expires
Attack #2: POST /execute-confirm twice with same token → Even with token burn, clock skew / retry logic could fool validation → Nonce tracking
prevents all retries
SOLUTION
1. Every POST to state-changing endpoints MUST include: { request_timestamp: “2026-03-30T15:45:30.123Z”, request_nonce: “uuid or random
string” }
2. Gateway validates:
Timestamp is not older than 5 minutes
Timestamp is not in future (with 10s clock skew tolerance)
Nonce has never been seen before
Store nonce + mark as used
3. Automatic cleanup
Nonces expire after 24 hours
Expired nonces removed every hour
SECURITY GUARANTEE

Same POST request cannot be executed twice, even if:

Attacker replays exact same request
Client retries due to network timeout
Attacker captures request in flight and replays later
Nonce is unique per request. Timestamp prevents old requests.
PRODUCTION NOTE

Currently in-memory. For high-scale production, upgrade to Redis with TTL-based expiry (code

ready, commented)
FIX #3: ED25519
REQUIRED (Mandatory Cryptographic Signatures)

Files: gateway/security/oauth-hardened.mjs gateway/routes/authorize-hardened.mjs
PROBLEM

ED25519_MODE defaulted to “optional”

This meant /authorize could be called without a signature: POST /authorize { intent_id: “abc-123”, decision: “approved”, authorized_by:
“brian.k.rasmussen” // NO signature required }
Attack: Attacker submits authorization without signature → Accepted if mode is “optional” → Brian’s private key never involved
SOLUTION
1. ED25519_MODE now defaults to “required” Environment variable: ED25519_MODE=required
2. /authorize endpoint now requires Ed25519 signature: POST /authorize { intent_id: “abc-123”, decision: “approved”, authorized_by:
“brian.k.rasmussen”, signature: “hex-encoded-ed25519-signature”, signature_timestamp: “2026-03-30T15:45:30Z” }
3. Gateway verifies:

Build canonical payload: { intent_id, action, decision, signer_id, timestamp }
Load signer’s public key from registered_signers
Verify signature against payload + public key
If valid: authorization approved
If invalid: authorization rejected

(FAIL CLOSED)

4. Key management:
Brian’s keypair generated once (Ed25519)
Private key held by Brian (not on gateway)
Public key registered in authorized_signers table
Signature proves Brian specifically approved this action
SECURITY GUARANTEE

Only someone with Brian’s private key can approve actions. No unsigned approval

allowed. Signature is cryptographic proof of intent and timestamp.
PRODUCTION NOTE
Keypair generation: gateway/security/ed25519.mjs already exists
Multi-signer support: Can register other approvers with their public keys
Key rotation: Supported (re-register new public key)
FIX #4: INTENT
PERSISTENCE (TODO — Low Priority)

File: TODO — gateway/governance/intents.mjs
PROBLEM

Intent objects stored in memory only. If gateway crashes, in-progress intents lost.

Scenario: Agent is in /executing state (token issued)
1. Gateway crashes
2. Gateway restarts
3. Token object lost from memory
4. Agent’s /execute-confirm fails (token not found)
5. Agent must resubmit intent and request new token
SOLUTION
1. Persist intent objects to PostgreSQL intents table (Schema already exists in init.sql)
2. On startup, load all intents from database into memory Sync database with in-memory cache on every update
3. Agent can retry with same token (persisted) No need to resubmit intent
EFFORT

~1 hour of work Straightforward database integration No API changes required

This is NOT a critical security fix, but improves operational resilience.
